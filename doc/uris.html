Boing: URI guide

Platform
O = Mac OS X, L = Linux, W = Windows1

Mode
I = Input, O = Output, F = Function

  URI query keys2
Node URIs
Data Redirection
OLW IO <in|out>:bridge:[//<HOST>:<PORT>]   Listen/Forward data on the default bridge (i.e. json.udp://:7898)3
OLW IO <in|out>:[<ENCODINGS>.]<IODEVICE>   Data bridge using custom input device and decodings34
OLW I play[.<ENCODING>]://<FILEPATH> loop, speed, interval Replay data from log file (default json)
OLW O log[.<ENCODING>]://<FILEPATH>   Record data to log file (default json)
Data Processing
OLW IO nop:   No operation node
OLW O dump<<OUTPUT-DEVICE>|:> request, filter, src, dest, depth Dump products to an output device (default stdout:)
OLW O stat<<OUTPUT-DEVICE>|:> request, filter, hz Print products statistics to an output device (default stdout)
OLW O viz: antialiasing, fps Display multi-touch contacts
OLW F filter:[<QUERY>]   Discard the products that do not match 'query'
OLW F filterout:[<QUERY>]   Discard the products that match 'query'
OLW F lag:[<MSEC>]   Add a lag to each received product
OLW F timekeeper:   Mark each received product with a timetag
OLW F calib: matrix, screen, attr, args,
request, resultmode		 Apply a 4x4 transformation matrix
OLW F filtering: uri, attr, args,
request, resultmode		 Filter product data using filtering library
IO devices
OLW I stdin:   Standard input
OLW O stdout:   Standard output
OLW I [file://]<ABSOLUTE-FILEPATH> uncompress, postend Absolute filepath (Read Only)
OLW O [file://]<ABSOLUTE-FILEPATH>   Absolute filepath (Write Only)
OLW I <RELATIVE-FILEPATH> uncompress, postend Relative filepath (Read Only)5
OLW O <RELATIVE-FILEPATH>   Relative filepath (Write Only)5
OLW I udp://<HOST>:<PORT>   Read from UDP socket
OLW O udp://<HOST>:<PORT> writeend Write to UDP socket
OLW IO tcp://<HOST>:<PORT>   Read/Write on TCP socket
Encodings
OLW IO slip   Bytestream from/to SLIP
OLW I json noslip JSON to products
OLW O json request, filter, noslip Products to JSON
OLW IO osc rt, noslip Bytestream from/to OSC
OLW IO tuio[.osc]   Multi-touch events from/to TUIO
L I mtdev   Multi-touch events from mtdev device6

Host
OLW I empty same as any address IPv6
OLW I 0.0.0.0 IPv4 any address
OLW I [::] IPv6 any address
OLW IO 127.0.0.1 IPv4 loopback
OLW IO [::1] IPv6 loopback
OLW IO x.x.x.x specific IPv4 address
OLW IO [x:x:x:x:x:x:x:x] specific IPv6 address
OLW IO hostname specific hostname

1On Windows, in order to define a file using the scheme file it is necessary to place the character '/' (slash) before the drive letter.
(e.g. file:///C:/Windows/explorer.exe)

2 The available query keys are obtained from the union of the available query keys of all the uri components. As an example, the uri out:bridge: is by default translated to out:json.udp://[::1]:7898, so it owns the query keys of the json encoder (request, filter) and of the udp socket node (writeend). Moreover, all input nodes also have the query keys pre, pre1, pre2... and all output nodes also have the query keys post, post1, post2... All function nodes have both of them.

3in: and out: prefixes can be omitted when using boing command line script, since it is obvious from the argument option.

4Some encodings have default input/output devices (e.g. in:tuio: is by default translated into in:tuio.udp://[::]:3333).

5It cannot be used to form composed URLs.

6mtdev decoding only works on linux device files (e.g. /dev/input/event6).
